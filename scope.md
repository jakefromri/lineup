# lineup — Scope

## One-Liner
A mobile-first scheduling app where team managers publish training sessions and announcements, and parents — via a shared link, no account required — view the calendar and mark their kids' attendance.

## Problem
Youth sports/activity teams coordinate schedules through group texts and spreadsheets, which gets messy fast. Managers need a simple way to publish session details and announcements; parents need a frictionless way to see what's coming up and say whether their kid will be there — without creating an account.

## MVP Scope
- Manager creates a team and gets a shareable parent join link
- Parents open the join link, add their contact info (name, email/phone) and their kid(s) names — no password, no account creation
- Managers create/edit/delete training sessions: name, date, time, location
- Parents see a calendar of upcoming sessions, 4 weeks out by default, with pagination/scroll for further weeks
- For each session, parents mark each of their kids as attending / not attending
- Attendance is visible to all parents on the team (not just managers)
- Original manager can invite additional managers; all managers have equal permissions
- Managers can post rich-text announcements (no threading); each shows author + timestamp, visible to all parents
- Each team has an API key for programmatic session management (create/update/delete sessions), so a manager can hand a calendar/file to Claude and have it populate the schedule via the API

## Out of Scope (MVP)
- Email, SMS, or push notifications of any kind (no reminders, no digest)
- Parent accounts / authentication beyond the join-link token
- Threaded comments or replies on announcements
- Approval workflows for attendance changes
- A parent belonging to multiple teams with a single identity (cross-team parent accounts)
- Recurring/repeating session templates (each session created individually, including via API)
- File attachments on announcements
- SSO / OAuth login for managers
- Public superadmin self-service (teams are created by superadmin only, in v1 that's Jake)
- Superadmin read-only view of a team's roster and sessions (redundant while Jake is also the only team's manager — revisit for multi-team)
- Merging duplicate parent records created via re-registration
- Rate limiting / abuse protection on the join-link registration endpoint (acceptable for a privately-shared link with a single team)

## User Roles
| Role | Description |
|------|-------------|
| superadmin | The operator (Jake). Creates teams, can view/deactivate any team. In v1, used only to bootstrap Jake's own team, but the role exists from day one for future multi-team use. |
| manager (tenant_admin) | A Supabase Auth user belonging to one team via a `memberships` row. Can create/edit/delete sessions, post/edit/delete announcements, view the full roster (parents, kids, attendance), invite other managers, and regenerate the team's parent join link and API key. All managers have identical permissions — no sub-roles. |
| parent (user) | Identified by a long-lived access token issued when they complete join-link registration — **not** a Supabase Auth account. The token (stored client-side) scopes them to one team and to their own parent/kid records. Can register/edit their own kids and contact info, view the team calendar and announcements, and mark attendance for their own kids. Cannot see other parents' contact info, only attendance status. |

## Tenancy Model
- A "tenant" = a team. Every team-scoped table carries `tenant_id`.
- Tenant 0 = Jake's own team (slug: `internal`), created at setup — this is the real team lineup is built for.
- Each team has a unique slug (used in URLs) and a unique **join token** (used in the parent join link, e.g. `/join/{join_token}`). The join token is opaque and can be regenerated by a manager, which invalidates the old link (existing parents already registered keep access via their personal access token — see Auth below).
- Managers are Supabase Auth users tied to exactly one team via `memberships` (role: `manager`). A user cannot be a manager of more than one team in v1.
- Parents are **not** Supabase Auth users. A parent record is created on join-link completion and is issued a random, unguessable `access_token`, stored in the browser (localStorage). This token is the parent's only credential — anyone with the token can act as that parent for that team. Losing it means re-registering (see Explicit Behaviors for recovery).
- All team-scoped data (sessions, announcements, parents, kids, attendance, API keys) carries `tenant_id` and is isolated per team. Teams have no visibility into each other.
- Deactivated teams: managers and parents see a "this team is no longer active" message; data is retained but inaccessible.

## Admin Panel

### Superadmin capabilities
- View all teams (name, slug, status, manager count, parent count)
- Create a new team — **atomically** creates the team, generates a join token, and creates the first manager invite in one operation; a team with no manager is not a valid state. Returns both the manager invite URL and the parent join URL.
- Deactivate / reactivate a team

> Superadmin read-only access to a team's roster/sessions is **deferred to Future Scope** (see Out of Scope) — in v1 there is one team and Jake is also its manager, so this view is redundant. Revisit if/when multiple teams exist.

### Manager capabilities
- View and manage the team roster: parents, their kids, contact info, and attendance history
- Create, edit, delete training sessions (name, date, time, location)
- Post, edit, delete announcements (rich text, author + timestamp)
- Invite additional managers (generates a manager invite link/token, accepted via Supabase Auth signup)
- View, regenerate, and revoke the team's parent join link
- View, regenerate, and revoke the team's API key (used for programmatic session management)
- Remove a manager from the team (cannot remove the last remaining manager — a team must always have at least one)

## Explicit Behaviors

### The system will:
- Allow anyone with a valid, non-revoked join link to register as a parent for that team — no approval step
- Allow a single parent record to register multiple kids during or after initial registration
- Issue each parent a random `access_token` on registration; this token (passed as a header or stored client-side) is required for all parent-scoped reads/writes
- Allow a parent who has lost their access token to re-register via the join link, creating a **new** parent record (old record and its attendance history remain but become orphaned — no merge in v1)
- Display the calendar to parents starting from "today," showing the next 4 weeks by default, with pagination (or scroll) to load further weeks/months
- Default a session's attendance status to "no response" for each kid until a parent explicitly marks attending/not attending
- Show attendance status (attending / not attending / no response) for every kid on the team, for every upcoming and past session, to all parents on that team
- Allow a parent to change their kid's attendance status at any time, including for past sessions
- Allow managers to edit or delete a session at any time; deleting a session also deletes its attendance records
- Require at least one **accepted** manager per team at all times — removing the last accepted manager is rejected (a manager may remove themselves only if at least one other accepted manager remains)
- Allow any manager to invite, view, and remove other managers (no manager hierarchy — see Open Questions/risk note)
- Grant the API key both read and write access to sessions (`GET`/`POST`/`PATCH`/`DELETE` on session resources) for the team it belongs to, and no access to any other resource (announcements, roster, attendance detail beyond what's embedded in session reads, team settings)
- Regenerating a join link or API key immediately invalidates the old one; already-registered parents are unaffected (their `access_token` is separate from the join token)
- Use `tenant_id` resolved from the authenticated context (manager JWT `app_metadata.tenant_id`, parent `access_token` lookup, or API key lookup) for all data operations — never accept `tenant_id` from the request body or query params
- Validate that the resolved auth context type matches the role(s) required by the endpoint (manager-only, parent-only, manager-or-parent, manager-or-apikey, superadmin-only); a mismatched context type returns `403`
- Redirect unauthenticated manager routes to a manager login page
- Show a clear "team is no longer active" state to managers and parents of a deactivated team, rather than a generic error — this applies to API-key requests as well (a deactivated team's API key returns the inactive-team error, not normal session data)
- Enforce a minimum password length of 10 characters for manager signup/password change
- Reject announcement create/edit requests with an empty or whitespace-only body
- Store and display session `date`/`time` as local wall-clock values with no timezone conversion (single-location-team assumption for v1; revisit if a team ever spans timezones)
- When a parent removes one of their kids, **archive** the kid rather than hard-deleting: the kid is hidden from the parent's active kid list and from attendance prompts on future sessions, but existing attendance records for that kid remain visible (with the kid's name) to managers and other parents on past and already-scheduled sessions

### The system will NOT:
- Send any email, SMS, or push notification to parents or managers
- Require parents to create a password or maintain a login session beyond their access token
- Allow a parent to view another parent's contact info (name/email/phone)
- Allow a parent to modify another parent's kids or attendance
- Allow the API key to access or modify announcements, roster, attendance management, or team settings
- Allow a manager to belong to more than one team
- Support threaded replies, reactions, or comments on announcements
- Auto-expire or archive old sessions or announcements — all history remains visible
- Merge duplicate parent records created via re-registration
- Expose the API key's stored hash (`api_keys.key_hash`) to any client, including managers — only existence/created/revoked metadata is returned via the API. If direct-from-client database access is ever introduced, `api_keys` must be accessed through a view that excludes `key_hash`.

## Open Questions
- Manager invite delivery: in v1, is the invite link shared manually by Jake (copy/paste), or does the system send an email via Supabase? → **Resolved for MVP**: manual sharing only — API returns the invite URL, no email is sent (consistent with "no notifications" constraint).
- Parent join link distribution: same as above — manual sharing (text/group chat), no in-app invite mechanism beyond generating the link.
- Should there be any limit on number of kids per parent or sessions per team? → **TBD**, default to no hard limit for MVP; revisit if abuse becomes relevant (it won't for a single team in v1).
- **Accepted risk (no manager hierarchy)**: any manager can remove any other manager, including all co-managers, leaving themselves in sole control. For v1 this is acceptable — managers are a small group of trusted people (e.g., coaches/team parents) Jake adds personally. Flagged here for awareness; revisit with a hierarchy/ownership concept only if abuse becomes a real concern.
